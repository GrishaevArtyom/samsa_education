## Организация работы с репозиторием (Git Workflow)

Данный раздел описывает правила взаимодействия с Git-репозиторием, принятые в команде.

### 1. Ветвление (Branching Strategy)
Мы используем упрощенную модель **Feature Branch Workflow**:

*   `main` — основная ветка. Содержит только стабильный, протестированный код. Ветки в `main` мержатся только через Pull Request.
*   `develop` — ветка разработки. Сюда сливаются готовые функции перед релизом.
*   `feature/<название-задачи>` — личные ветки участников. Создаются от `develop` для реализации конкретной задачи.
    *   *Пример:* `feature/login-form`, `feature/database-connection`.

### 2. Коммиты (Commit Conventions)
Стараемся писать осмысленные сообщения коммитов.
Формат: `<тип>: <описание>`

**Типы коммитов:**
*   `feat` — новая функциональность.
*   `fix` — исправление ошибки.
*   `docs` — изменение документации.
*   `style` — форматирование (пробелы, запятые), не влияющее на логику.
*   `refactor` — рефакторинг кода без изменения функционала.
*   `test` — добавление или изменение тестов.

*Пример:* `feat: добавлена авторизация через Google`

### 3. Слияние кода (Merge & Code Review)
1.  Запрещен прямой пуш в ветки `main` и `develop`.
2.  Любые изменения вливаются через **Pull Request (PR)**.
3.  Перед созданием PR необходимо:
    *   Убедиться, что код компилируется/запускается.
    *   Сделать `rebase` или `merge` актуальной ветки `develop` в свою функциональную ветку, чтобы избежать конфликтов.
4.  Каждый PR должен быть проверен (Code Review) как минимум одним другим участником команды.
5.  Ветка удаляется после успешного мержа.

### 4. Управление задачами
Задачи ведутся в разделе **Issues** данного репозитория.
*   Каждая задача имеет статус (Open, In Progress, Done).
*   Ветка называется в соответствии с номером задачи (если возможно), например: `feature/issue-12-login`.
